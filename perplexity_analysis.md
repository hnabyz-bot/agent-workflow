1. 현재 합의된 현실 전제
Agent들끼리는 직접 대화·상태 공유가 불가능하고, 모든 협업은 사람이 중개해야 한다.

따라서 “자동 협업”이나 “실시간 피드백”이 아니라, 단계별로 한 Agent씩 호출하고 사람이 요약/복붙으로 이어주는 구조가 현실적이다.

그럼에도 불구하고, 적절한 지점에서 다른 Agent를 끼워 넣으면 순수 단일 Agent 사용보다 품질·안정성 측면에서 이득이 있다(시간 오버헤드는 있지만 감당 가능한 수준).

2. Phase 구조와 역할 분담에 대한 전문가 의견
유지할 큰 틀
Phase 0–3 구조는 그대로 유지할 가치가 있다.

Phase 0: Copilot 중심의 PoC로 빠른 기술 검증.

Phase 1: Claude 중심의 계획·요구사항 정리, 필요할 때만 Gemini로 기술 검토/대안 설계.

Phase 2: Copilot 주 구현 + Gemini 설계/분석 + Claude 테스트 skeleton.

Phase 3: Copilot DevOps·배포 스크립트, Claude 문서화, Gemini 최종 기술 검수.

표현·운영 측면 조정
“협업” 표현 정리

“3 Agents가 함께 검토한다 / 실시간 피드백 루프” 같은 표현은
→ “사용자가 이전 결과를 요약해서 다른 Agent에 단계별로 전달한다”로 바꿔야 한다.

즉, **협업의 단위는 ‘사람이 Agent를 순차 호출하는 시퀀스’**임을 명시.

역할 비율(%)의 의미 축소

“Gemini 30%, Copilot 40%, Claude 20%” 같은 수치는 정밀 측정값이 아니라 작업 분배 감각을 주는 가이드로만 사용해야 한다.

문서에는 “절대적인 수치 X, 어디에 더 시간을 쓸지 방향성을 주는 참고값”이라는 주석을 붙이는 것이 적절하다.

Sprint → 마일스톤/반복 사이클로 재정의

Agent 기준 Sprint는 컨텍스트 한계를 고려할 때 의미가 약하다.

대신, 사람/팀 기준 “기능 단위 마일스톤 반복(계획 → 구현 → 검증 → 정리)”으로 표현하고,

각 반복 시 Agent에게는 요약 문서 + 주요 변경점 + Git 히스토리만 전달하는 식으로 현실화하는 것이 좋다.

3. 팀 규모 기준 운영 전략
1인 개발자

기본은 거의 순차 활용:

구현: Copilot

막혔을 때: Gemini (알고리즘/설계 질문)

마무리 문서: Claude

Phase 0/1을 크게 형식적으로 가져가기보다는, 필요할 때만 “간단 PoC → 최소 계획 → 바로 구현” 식으로 간소화.

2–5인 팀

기존 문서의 Phase 0–3를 그대로 쓰되, 각 Phase별로 “어떤 사람이 어떤 Agent를 담당하는지”를 명확히 나눠야 한다.

예:

A: Claude 담당(계획/문서)

B: Copilot 담당(구현/실행)

C: Gemini 담당(리뷰/분석)

이 구조에서 사람이 Agent 간 허브 역할을 하므로, 과도한 왕복만 피하면 협업 워크플로우는 실현 가능하다.

5인 이상 팀

원래 워크플로우를 더 적극적으로 활용할 수 있지만, 여전히 전제는

Agent 간 자동 통신 X

문서/요약/이슈 트래킹을 통한 사람 간 릴레이 O

즉, Agent는 팀의 “특화된 도구”이고, 링크를 이어주는 것은 항상 사람이라는 점을 분명히 해야 한다.

4. 구체적인 운용 규칙(각 Agent가 의식해야 할 부분)
한 번에 한 Agent 원칙

기본적으로 어떤 Task/Phase에서는 한 Agent가 주도자가 된다.

다른 Agent는 “막혔을 때” 혹은 “고난도 설계/리뷰” 같은 특정 지점에서만 호출된다.

요약 기반 컨텍스트 전달

Agent에게는 항상 “지금까지 진행 상황 요약 + 현재 질문/목표”만 간결하게 전달한다.

전체 대화를 계속 붙여넣는 것이 아니라, 사람이 만든 요약을 중심으로 누적 컨텍스트를 유지한다.

역할 책임 분리

Claude: 요구사항·계획·문서·테스트 시나리오의 언어적/구조적 정리에 집중.

Gemini: 알고리즘·아키텍처·성능·보안 등 고난도 기술 판단에 집중.

Copilot: 파일 편집, 코드 구현, 터미널/빌드/테스트, Git 등 실행/자동화 작업에 집중.

실패/에러 처리 프로토콜

빌드/테스트/PoC 실패의 1차 Owner는 항상 Copilot.

Copilot이 원인이 설계·요구사항이라 판단되면,

설계/알고리즘 문제: Gemini로 escalate

요구사항/계획/문서 문제: Claude로 escalate

5. 각 Agent에게 바라는 추가 액션
각 Agent에게 요청할 수 있는 질문 예시는 다음과 같다.

Claude에게:

“위 전제를 반영해서, 계획/문서/테스트 skeleton 쪽 가이드를 다시 정리해줘. 특히 요약 기반 컨텍스트 전달과 비율(%)를 가이드로만 쓰는 규칙을 문서 상단에 명시해줘.”

Gemini에게:

“위 구조에서 네 역할을 **‘고난도 설계/분석 + 테스트 전략 정의’**로 더 선명하게 만들고, 언제 너를 호출하는 것이 가장 효과적인지(예: 어떤 종류의 이슈/알고리즘에서) 예시를 들어 정리해줘.”

Copilot에게:

“위 구조를 기준으로, 실제 개발/빌드/테스트/배포 단계에서 어떤 명령/작업 단위를 표준화하면 좋은지(예: git, 테스트, 빌드 스크립트 패턴)를 체크리스트로 만들어줘.”

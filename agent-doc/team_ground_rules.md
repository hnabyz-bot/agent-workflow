실제 팀 적용용 운영 규칙
1) 공통 운영 원칙
모든 작업은 “어느 Phase, 어느 Activity를 수행 중인지”를 명시하고 시작한다.
​

각 Activity에는 명시적인 Owner(R/A)가 있어야 하며, Owner가 바뀔 때마다 간단한 Hand-off 메모(요약)를 남긴다.
​

2) Agent 호출/사용 규칙
설계/아키텍처/성능·보안 판단이 필요하면 ⇒ 우선 Gemini 호출, 결과를 Claude 문서에 반영 후 Copilot 구현으로 넘긴다.
​

대량 코드 수정/테스트 실행/빌드/배포/스크립트 작업 ⇒ Copilot을 기본 진입점으로 사용한다.
​

요구사항 정리, 계획, 회고, 매뉴얼·운영 문서, 아키텍처 설명 ⇒ Claude를 기본 진입점으로 사용한다.
​

3) 실패 처리 규칙(Fail Policy)
실행/빌드/테스트 실패는 “무조건 Copilot이 1차 Owner”가 되어 로그를 수집하고, 원인이 설계/요구사항인 경우에만 Gemini/Claude로 escalate 한다.
​

설계적 모순·성능 한계가 확인되면, 해당 Sprint에서 무리하게 우회하지 말고 “다음 Sprint에서 설계 수정 Task를 우선순위 상향”하는 것을 기본 원칙으로 한다.
​

4) 테스트/품질 규칙
Gemini가 Sprint 시작 시 “이번 Sprint의 테스트 전략(커버리지 목표, 성능/장애 시나리오)”을 5줄 이내로 정의한다.
​

Claude는 이 전략에 맞춰 테스트 skeleton을 만들고, Copilot은 해당 테스트를 모두 통과시키는 것을 Sprint의 Done 조건으로 삼는다.
​

Sprint Review 시 “실패한 테스트, 미작성 테스트 목록”을 반드시 한 번 리뷰하고, 차기 Sprint에 반영한다.
​

5) 문서–코드 동기화 규칙
매 Sprint 종료 시점에 Copilot은 “이번 Sprint에서 변경된 주요 API/모듈/설정”을 리스트로 요약해서 Claude에 전달한다.
​

Claude는 이 요약과 실제 코드 기준으로 문서(설계, 운영 가이드, API 명세)를 갱신하는 것을 Sprint 종료의 필수 조건으로 한다.
​

6) FPGA/도메인 특화 규칙(옵션)
HDL/FPGA 관련 변경 시에는 “인터페이스/타이밍/리소스 영향 분석”을 Gemini에게 먼저 요청하고, 그 결과를 Claude가 명세서에 기록한 뒤 Copilot이 테스트벤치/빌드 스크립트로 구현한다.
​

HW/SW 경계(예: 레지스터 맵, AXI 인터페이스) 변경은 항상 문서–테스트–코드가 동시에 갱신되도록, 하나의 Task 안에서 Claude–Copilot–Gemini 순서로 한 번에 처리한다.
​
